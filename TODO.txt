+ Let's try a different block counting algorithm. It should work like this
1. make a list of all parcels in the map viewport
2. select the first parcel that is not a road. Add block:1 property to it.
3. use a floodfill type of algorithm to select a geographically neighbouring parcel (that is also on the list and not a road) and make it part of the same block (add block:1 to its properties). Repeat the process recursivelly until you run out of neighbouring parcels to select. That is our first block. To find the next block, find the next parcel on the list that is both not a road and also does not have the block property set yet. Repeat the floodfill algorithm. Repeat this process of finding blocks until you have parsed the entire list of parcels in the map viewport.

Rules for splitting parcels

Whenever we split a parcel:
- if parcel number is an integer (2036), we need to parse all parcels to find the last subnumber (2036/3, for example). If there are no subnumbers, the next number to assign is 2036/1, and if there is more than one part then in +1 increments they are 2036/2, 2036/3, etc.
If there are subnumbers, then the new part, if one, is the last subnumber +1, and if multiple, then +1 for each following.

We shoud not modify parcels we load from server by keeping the cestica_id and changing geometry or similar.

1. Modifying a parcel (by changing geometry or splitting into multiple parcels) should always create a parcel(s) with a new parcel_numbers and cestica_ids
2. Each of the new parcels needs to keep a reference to the parcel that is its parent parcel (can have a maximum of 1 parent parcel)
3. The parcel number is formed by finding the root number (before slash), then finding the last used subnumber and increasing it by one (if no subnumbers are found then last is 0)
4. The cestica_id can be formed by using the root number parcel's cestica_id and adding subnumber after underscore. So if 2036 had id 21462649 the children would be 21462649_1, 21462649_2, etc.
5. The same applies for the newly created road parcel
6. If a parcel is not just having modified geometry, but is split into multiple parts, the parts shuld all be new parcels and follow the rules above
7. The parent parcel should be removed from the map so that it is not drawn when the map is redrawn, but needs to be preserved in memory or storage so that we can bring it back/restore it when/if we roll back proposals
8. If the parcel 2036/2, that was created when 2036 was split into two, is split into two again, the two new parcels should get numbers 2036/3 and 2036/4, but their parent needs to be 2036/2 (not 2036), even though we will remove it from the map. (NOTE: is there a good reason for this? Why not just have 2036 as parent?)
9. When loading parcels from server/source and preparing to draw them, we need to skip drawing and remove from the set those that have child parcels in local storage. The way to figure out those is by maintaining an array of cestica_id in local storage where we add a cestica_id whenever it is modified locally. The local new parcels need to be merged into the set.

--------- Refactor
CESTICA_ID and PARCEL_NUMBER are currently different in the following sense. The CESTICA_ID is a database id, a simple unique integer used accross many systems. PARCEL_NUMBER is a legacy, human-readable number that also appears in all the government registries and court documents and is used by people on day to day basis. It is not unique, to get croatia-level unique parcel identifier it has to be PARCEL_NUMBER+CADASTRAL_MUNICIPALITY.
That is why in UI we are often showing PARCEL_NUMBER. CESTICA_ID can't really be used by humans, just by systems. For this reason we should probably first refactor the frontend to get rid of cestica id and replace it with a parcel_id that is in the format "PARCEL_NUMBER-CADASTRAL_MUNICIPALITY"
we could map CESTICA_ID to PARCEL_ID_2, because it is sometimes good to show it at the frontend for debugging purposes. Also this situation of dual keys we may incounter in other systems too. But we should use PARCEL_ID_2 only for displaying (in specific cases, in debug mode for example) and never for API calls and data organization. The backend APIs need to be reworked to take PARCEL_ID and work off that.
Only once we have standardized the current frontend on PARCEL_ID should we build adapters to translate/convert incoming data to PARCEL_ID.
Note that, with parcel_id becoming human-readable, we don't need to use parcel_number for display. We should rename it to parcel_id_short and it should be just the first part of parcel_id for Zagreb (the xxx in xxxx-yyyyy), and just the last part of parcel_id for Buenos Aires (the zzz in xxx-yyy-zzz). The short version should be used wherever the parcel_number is used now.
