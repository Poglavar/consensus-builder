Difference in market value (potential profit): current market value subtracted from new market value. How to calculate the values:
  1) parcel area
  current: area of the parcel
  new: area of the parcel (should be same for building proposals and different if the proposal is a road splitting or changing the parcel)
  2) building footprint
  current: the area of the building(s) on the parcel currently
  new: area of the part of the block building polygon that intersects the parcel
  3) building height
  current: put current height, which we assume to be 10 meters
  new: depends on the height of the proposed building, pull from there
  4) building floors
  current: current height / 3, round down
  new: proposed height / 3, round down
  5) square meters
  current: current footprint x current floors
  new: new footprint x new floors
  6) parking spots
  current: 4
  new: 0
  7) estimated market value of the building
  current: current square meters x square meter price in the area (assume 3500 EUR for now)
  new: new square meters x square meter price in the area (assume 3500 EUR for now)

+ We need to get rid of the parcel sidecar keys. A parcel can be an original/base map parcel, or it can be a descendant of a proposal (maximum one proposal). It can also have a proposal that descends from it (also a maximum of one). Only the proposal that has been Applied has descended from the parcel. The other proposals that are made but not Applied are not considered parcel descendants. Whenever a proposal is applied and includes/affects a parcel, it:
  - makes the parcel one of its ancestors (a proposal can have multiple ancestor parcels)
  - makes itself the parcel's descendantProposal
  - optionally: creates one or more descendant parcels from that parcel, to which descendant parcels it becomes the ancestorProposal. Most proposals create descendant parcels (road/track for example), but some do do not (urban rule proposals).
We should store the two proposal variables on each parcel object: ancestorProposal, descendantProposal. Both can exist or not, be empty or set. Not existing = not set.
When a proposal is unapplied, it has to clear itself from descendantProposal of all the ancestor parcels. Let's make sure all proposal types set and unset the two variables when applied/unapplied. This set/unset both happen to ancestor parcels. To descendant parcels only set happens, but unset no, because they are destroyed on proposal being unapplied.

Parcels do not (directly) descend from parcels, but from proposals. Proposals do not (directly) descend from proposals, but from parcels.
Starting from a parcel we can traverse "up" the graph to find all the proposals that have been applied in order for the parcel to be created. This is useful. When sharing a proposal that affects (has ancestors) parcels that have been descended from other proposals, we would need to include those ancestor proposals in the share too.
We can also traverse down to see which proposal descends from the parcel. Unapplying a proposal would include recursively traversing its descendant parcels to see if they have any descendant proposals that need to be reversed first.
